1 Write Verilog

Create/edit your .v files in VS Code.

# Your Verilog goes in rtl/
code rtl/mac_unit.v
Tip: Keep the Verilog reference (docs/verilog_reference.html) open in a browser tab while coding.

2 Lint (Verilator)

Catches syntax errors, width mismatches, unsigned/signed issues, unused signals. Run this BEFORE simulating.

# Lint a single file
verilator --lint-only -Wall rtl/mac_unit.v

# Lint multiple files
verilator --lint-only -Wall rtl/mac_unit.v rtl/mac_array.v
Success: No output = no errors. Silence is good.
Errors: Fix everything it complains about. Common ones:
• WIDTHEXPAND — assigning smaller signal to larger one (might need sign extension)
• UNSIGNED — mixing signed and unsigned signals
• UNUSED — declared a signal but never used it
• UNDRIVEN — using a signal that nothing drives

3 Simulate (Icarus Verilog + cocotb)

Runs your Verilog against the Python testbench. Shows PASS/FAIL for each test.

# Run all tests for a module
cd tb && make

# Run a specific test only
cd tb && make TESTCASE=test_basic_multiply

# Clean simulation files and rerun
cd tb && make clean && make
Success: All tests show PASS. Output ends with something like passed X of X tests.
Failure: Shows expected vs actual value. Go check your Verilog logic. Use waveforms (step 4) to debug.
What's happening: Icarus Verilog compiles your .v file into a simulation. cocotb (Python) drives the inputs and checks the outputs. The Makefile ties them together.

4 View Waveforms (Surfer)

Visual debugger. See every signal's value at every clock cycle. Use this when tests fail.

# Open the waveform file generated by simulation
surfer tb/sim_build/dump.fst

# If the file is .vcd instead
surfer tb/sim_build/mac_unit.vcd
How to use Surfer:
• Left panel: list of signals in your design
• Drag signals to the main view to see their waveforms
• Zoom in/out to see individual clock cycles
• Click on a time point to read exact signal values
• Look for where expected vs actual values diverge

5 Resource Estimate (Yosys)

Tells you how many LEs, multipliers, and memory blocks your design uses. Quick budget check.

# Basic resource count
yosys -p "read_verilog rtl/mac_unit.v; synth; stat"

# Quieter output (just the stats)
yosys -p "read_verilog rtl/mac_unit.v; synth; stat" 2>&1 | grep -A 20 "Printing statistics"

# Multiple files
yosys -p "read_verilog rtl/mac_unit.v rtl/mac_array.v; synth; stat"
What to look for in output:
• $mul — multiplier blocks used
• $add — adder cells
• Number of cells — rough LE estimate
• SB_DFF or DFF — flip-flops (registers)

Our budget: 114,480 LEs, 266 multiplier blocks, 432 M9K BRAM blocks

6 Git Commit

Only after the module passes all tests.

# Check what changed
git status
git diff

# Stage and commit
git add rtl/mac_unit.v tb/test_mac_unit.py tb/Makefile
git commit -m "feat: add MAC unit with passing testbench"

# See commit history
git log --oneline
